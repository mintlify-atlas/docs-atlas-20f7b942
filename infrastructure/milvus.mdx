Milvus is the vector database that stores dense embeddings of medical document chunks. It powers the semantic retrieval channel of the agentic pipeline, enabling hybrid search over both unstructured text and graph context. By maintaining a dedicated vector index alongside the Neo4j knowledge graph, Milvus gives the system a high-performance store for similarity-based retrieval that complements the structured SPO triple queries issued by the relational channel.

## Role in the System

Milvus participates in both the indexing and retrieval stages of the agentic pipeline.

**During indexing**, each document chunk is embedded into a dense vector representation and written into Milvus together with its raw text. This happens in parallel with the entity and relationship extraction that populates Neo4j, so the semantic and relational indexes are built in a single ingestion pass.

**During retrieval**, the semantic channel performs dense vector search over Milvus to identify the document chunks most relevant to each sub-query generated by the decomposition step. The top-ranked chunks are then filtered, summarised, and merged with the relational channel's output before the final answer is synthesised.

**LightRAG integration** — LightRAG stores its chunk embeddings in a Milvus-backed vector index that sits alongside the Neo4j knowledge graph. This dual-store arrangement is what allows LightRAG to operate in hybrid mode, combining keyword-based knowledge graph traversal with dense vector search over document chunks in a single retrieval call.

## Setup

The quickest way to run Milvus locally is in standalone mode via Docker.

<Steps>
  <Step title="Start the Milvus standalone container">
    Pull and start the Milvus standalone image with the default gRPC and HTTP ports exposed:

    ```bash
    docker run -d --name milvus-standalone \
      -p 19530:19530 -p 9091:9091 \
      milvusdb/milvus:latest standalone
    ```
  </Step>

  <Step title="Verify the instance">
    Once the container is running, confirm it is healthy by checking the HTTP health endpoint:

    ```
    http://localhost:9091/healthz
    ```

    The application connects over gRPC on the default port:

    | Protocol | Port |
    |---|---|
    | gRPC (primary) | `19530` |
    | HTTP / health | `9091` |
  </Step>
</Steps>

<Tip>
  For production deployments, Milvus can be deployed on Kubernetes using the official [Milvus Helm chart](https://milvus.io/docs/install_cluster-helm.md), or as a fully managed service via [Zilliz Cloud](https://zilliz.com/cloud).
</Tip>

## Configuration

The pipeline connects to Milvus using the following environment variables. Set these in your shell or in a `.env` file and never hardcode them in source code.

| Variable | Description | Example |
|---|---|---|
| `MILVUS_HOST` | Hostname or IP address of the Milvus instance | `localhost` |
| `MILVUS_PORT` | gRPC port for the Milvus instance | `19530` |

## Hybrid Search

Milvus supports hybrid retrieval combining dense and sparse vector strategies. The agentic pipeline — specifically LightRAG operating in hybrid mode — exploits this capability to execute two complementary retrieval operations in a single pass:

1. **Keyword-based KG traversal** — a sparse, graph-structured search that surfaces entities and relationships from the Neo4j knowledge graph that are lexically relevant to the query.
2. **Dense vector search** — a similarity search over the Milvus-backed chunk embeddings that retrieves semantically related document passages even when there is no exact keyword overlap.

The results of both operations are merged by LightRAG's context builder into a structured JSON response containing four sections: `Knowledge Graph Data (Entity)`, `Knowledge Graph Data (Relationship)`, `Document Chunks`, and a `Reference Document List`. The pipeline's `context_filter` then splits this response, routing the document chunks and reference list to the semantic channel and the entity and relationship JSON to the relational channel.
